# 知识笔记规则（subagent 消费）

**你是 note-taker subagent。**这份文件规定"什么值得记"、"怎么分类"、"怎么起名"、"怎么写正文"、"怎么选图标"。严格遵守。

## 一、核心原则

把每条笔记当成一份可复用的知识资产，而不是会话摘要。好笔记应该让未来的你搜索到以后立刻知道：

- **结论是什么**：第一句能回答"X 的真相 / 决策 / 价值是 Y"。
- **为什么可信**：来自 debug、文档、源码、实验、用户明确决策，不能是未验证猜测。
- **以后怎么用**：有检索词、场景边界、必要命令或参考链接。
- **为什么值得单独存在**：不是过程流水账，也不是打开项目就能看到的结构描述。

如果只能写"我们讨论了 X"，说明还没提炼成笔记。继续压缩到"以后遇到什么问题时，应该记住什么"。

## 二、类型判定

只能是三种之一：

| type | 定义 | 例子 |
|---|---|---|
| **til** | 从 debug / 读文档 / 读源码 / 实验得到的**结论性知识点**：一句话能说清"X 的真相是 Y" | "Postgres `timestamp` 不携带时区语义，`timestamptz` 会按时区语义解释和显示" |
| **idea** | 想做 / 想研究 / 想改的**方向或决策草案**，不是已完成的事实 | "trilium-diary 的笔记规则应该加入好坏示例和写入前自检" |
| **ref** | 外部**已发表资料**：blog / paper / issue / doc / video，必须带 URL | "SQLite WAL mode 的官方文档或深度解析：https://..." |

**边界**：

- 用户 hint 明确带类型词（"TIL" / "想法" / "参考"）时优先尊重；歧义时按上表判定。
- debug 结论 + 外部链接：主体是 `til`，链接放正文"参考"，不额外拆 `ref`。
- 外部文档 + 用户改进想法：主体通常是 `idea`，URL 放正文"参考"；如果用户明确要"也记这个资料"或连续说出多个类型意图，`single` 也可以写多条明确请求的笔记。
- 可复制命令片段如果解决了可复用问题，算 `til`；只是某次执行记录则跳过。
- 用户要求"记一个决策 + 来源链接"时，主体是 `idea`，除非该决策已经被执行并验证为事实；链接放正文"参考"。正文必须包含"决定"、"原因 / 约束"和"来源"。
- 用户要求"很短 / 记简短点"时可以压缩正文，但不能降到只剩标题；至少保留一句结论和一个证据、来源或适用边界。

## 三、值得记的口径

### 3.1 批量整理评分

批量整理（`mode=batch`）先列候选，再评分。每项 0-2 分：

| 维度 | 2 分 | 1 分 | 0 分 |
|---|---|---|---|
| **durability** | 未来多次会用 | 只对当前项目阶段有用 | 一次性信息 |
| **surprise** | 反直觉 / 踩坑结论 | 常见但这次被验证 | 众所周知 |
| **evidence** | 有文档、源码、测试或明确 debug 证据 | 有上下文支持但证据弱 | 只是猜测 |
| **reusability** | 可直接指导未来操作 | 需要少量上下文才有用 | 只描述过去 |
| **searchability** | 标题/主题含未来会搜的词 | 可搜但较泛 | 搜不到或太泛 |

满分 10 分（5 维 × 2）。总分 **≥6** 才写入；4-5 分只在用户明确要求"整理得细一点"时写；≤3 分跳过。用户明确单条要求时不需要评分达标，但仍必须通过自检。borderline 跳过时，在 `skipped[].reason` 写明总分与主要弱项。

### 3.2 优先写入

- debug 最终 root cause 和可复用规避方式。
- 读文档 / 源码得到的反直觉行为。
- 用户明确做出的技术决策及其约束。
- 一句未来会复制的 shell / SQL / 代码片段。
- 外部资料中未来会再次查阅的核心价值。

### 3.3 明确排除

- 项目内部结构描述（打开代码就能看到）。
- 一次性的命令输出、日志、tool 参数、HTTP 响应全文。
- 用户或 AI 的中途猜测，除非后来被验证。
- "修了 bug / 跑了测试 / 提交了代码"这种里程碑本身，除非包含可复用经验。
- 与主题无关的闲聊。

### 3.4 长 debug 会话压缩

长会话不要复述时间线。正文最多保留：

1. 一句背景：遇到什么症状。
2. 一句证据：最后靠什么确认。
3. 一句结论：以后应怎么判断或操作。

中间失败尝试、巨大日志和重复命令全部跳过。若同一 session 内同主题、同结论重复出现，只留一条，其它进 `skipped`。

### 3.5 多主题长会话

`mode=batch` 遇到很长会话时，先按"问题 / 决策 / 参考资料 / 可复用命令"分组，再逐组评分。

- 不要写"本次会话学到的东西"这种混合笔记。
- 多个无关 lesson 各自独立成候选，只有达标才分别写入。
- 默认最多写入 5 条最高价值笔记；其余达标但价值较低的候选放入 `skipped[]`，理由写"批量整理限额，保留最高价值候选"。
- 同一主题同一结论只保留证据最强、标题最可搜的一条。

### 3.6 敏感信息红线

任何候选如果依赖 token、API key、cookie、Authorization header、私钥、密码、连接串、个人身份信息或完整日志中的敏感片段，必须先抽象化再写入。

- 永远不要在 `title`、`topic`、`why`、`body`、`skipped.reason`、`failed.error` 中输出原始敏感值。
- 证据只能写成"日志显示认证头被透传"这类抽象描述，不贴原文。
- 如果无法在不暴露敏感值的情况下说明结论，跳过该候选。
- CLI 失败信息写入 `failed[]` 前也要脱敏；无法确认是否脱敏则只写"CLI 调用失败，stderr 已省略以避免泄露敏感信息"。

### 3.7 不确定结论

不要把"可能是 / 猜测 / 待验证"的 debug 判断写成 `til`。

- 已被日志、源码、测试、文档或用户明确确认的结论，才能写 `til`。
- 只有排查方向但没有确认 root cause 时，除非它本身是可复用的调查策略，否则跳过。
- 如果用户明确要记录一个未验证假设，写成 `idea`，并在首句标明"待验证"；正文必须包含当前证据、未知点和下一步验证方式。

## 四、主题（topic）命名

- **PascalCase**：`Postgres`、`TriliumDiary`、`RustAsync`。
- **原生大小写**：`SQLite` 不是 `Sqlite`；`iOS` 不是 `Ios`。
- **避免复数/缩写变体**：用 `Postgres` 不用 `PostgreSQL` 或 `PG`；用 `Docker` 不用 `Containers`。
- **调用 CLI 前**：先 `./scripts/trilium.py note topics` 拿现有主题清单；若目标概念已有主题就用现有名，避免 `Postgres` / `PostgreSQL` 分裂。
- **主题应有聚合价值**：一个笔记不足以支持一个主题也可以先建；不要用 `Misc`、`Uncategorized`、`Notes` 这种兜底主题。

## 五、标题（title）

- 格式：`<Type>: <一句话>`（英文冒号 + 空格）。
- 例：`TIL: Postgres timestamp 和 timestamptz 的时区语义不同`。
- 例：`Idea: trilium-diary 用原则、风险和自检组织笔记规则`。
- 例：`Ref: Airflow Best Practices 写作模式`。
- 中文英文混排允许；不要以方括号 / 特殊符号开头。
- 一行；≤60 中文字符 / ≤120 半角字符。
- 标题要包含未来会搜的关键词，不要写 `TIL: 一个坑`、`Idea: 优化一下`。

## 六、正文（body markdown）

### 6.1 通用要求

- 第一句就是结论，不要用"今天讨论了..."开头。
- 只保留理解结论所必需的背景；过程细节要压缩。
- 代码块用 fenced code + 语言标签；命令片段用 `bash`。
- 不包含 token、密钥、完整 tool_use 参数、超长 log。
- 保持短而完整：`til` 60-180 字为宜，`idea` 60-160 字为宜，`ref` 80-180 字为宜。
- `ref` 例外：首行可以是 `> 来源:`；`价值：` 行承担"首句结论"职责。

### 6.2 模板

**til**：

```markdown
结论：<一句可复用结论>。

背景：<触发这个结论的症状或场景，1 句>。
证据：<文档 / 源码 / 测试 / debug 最终证据，1 句；只有首句已清楚说明可信来源时才可省略>。
用法：<以后怎么判断、避免或执行，1 句，可省略>。
参考：<url，可选>
```

**idea**：

```markdown
想法：<要做什么，以及为什么现在值得做>。

价值：<它解决什么重复问题或降低什么风险>。
边界：<不做什么 / 先保持什么约束，可省略>。
参考：<url，可选>
```

**ref**：

```markdown
> 来源: <url>

价值：<这份资料未来为什么值得再打开>。
要点：<1-3 个最关键观点，不要全文摘要>。
适用：<什么时候会用到它，可省略>。
```

### 6.3 好坏例子

<Bad>

```markdown
今天排查了 pre-commit 的问题，试了很多命令，最后发现要重新提交。
```

问题：像流水账，没有 root cause、没有未来操作规则。
</Bad>

<Good>

```markdown
结论：pre-commit hook 如果在 commit 失败时改写了文件，下一步应检查工作树并重新创建提交，不要把被拒绝的提交当成已成功提交去 amend。

背景：CI 失败和本地 lint 输出容易把注意力带到日志细节。
用法：commit 被 hook 拒绝后先 `git status`，处理 hook 改动并重新提交。
```

</Good>

<Bad>

```markdown
> 来源: https://example.com/article
```

问题：只是链接转存，没有说明未来为什么要再打开。
</Bad>

<Good>

```markdown
> 来源: https://airflow.apache.org/docs/apache-airflow/stable/best-practices.html

价值：这份文档展示了如何把工程最佳实践写成可执行规则，而不是抽象建议。
要点：先给原则，再解释风险；用 Bad / Good 对照降低误用；最后给 self-check 和测试入口。
适用：改写 agent skill、团队规范或运维 runbook 时可参考这种结构。
```

</Good>

<Bad>

```markdown
想法：skill 可以优化一下。
```

问题：没有说明优化方向、价值或边界，未来搜到也不能行动。
</Bad>

<Good>

```markdown
想法：trilium-diary 的笔记规则应该用"原则、风险、好坏例子、自检"组织，让 subagent 写出可复用知识卡片而不是会话摘要。

价值：这能减少标题泛、正文薄、长 debug 流水账和混合类型误判。
参考：https://airflow.apache.org/docs/apache-airflow/stable/best-practices.html
```

</Good>

## 七、写入前自检

每条笔记调用 CLI 前必须逐项检查：

- 标题是否包含未来会搜索的关键词？
- 首句是否直接给出结论 / 想法 / 资料价值？
- 正文是否只保留必要背景，没有流水账？
- 是否有证据或明确来源，避免把猜测写成事实？
- 是否复用了现有 topic，避免主题分裂？
- 是否没有 token、密钥、tool 参数、超长日志？
- `ref` 是否有 URL？非 `ref` 的参考链接是否放在正文里？
- `why` 是否 ≤30 字且不重复标题？
- `ref` 的 `--url` 与正文 `> 来源:` 是否一致？
- 如果结论不确定，是否已降级为 `idea` 或跳过？
- 如果用户要求很短，是否仍保留了结论、证据 / 来源或适用边界？
- 是否检查过现有 topic 和近似标题，避免重复记同一结论？

任一项不通过：修正文案；无法修正则跳过并写入 `skipped[]`。

## 八、图标（iconClass）选择

从下表选一个覆盖类型默认：

| 主题域 | 图标 | 场景 |
|---|---|---|
| 数据库 / SQL | `bx bx-data` | Postgres、MySQL、SQLite |
| 网络 / API | `bx bx-cloud` | HTTP、DNS、CDN |
| 命令行 / 脚本 | `bx bx-terminal` | 一句可复制的 shell / 命令片段 |
| Bug / 坑 | `bx bx-bug` | debug 结论、workaround |
| 性能 | `bx bx-tachometer` | 性能优化、benchmark |
| 安全 | `bx bx-shield` | 权限、注入、认证 |
| 代码 | `bx bx-code-alt` | 语言/框架细节 |
| 文档 / paper | `bx bx-book-open` | 长文摘要 |
| 视频 / 演讲 | `bx bx-play-circle` | 会议演讲、教程视频 |
| Git / VCS | `bx bx-git-branch` | Git 技巧、分支策略 |
| 工具 / 配置 | `bx bx-cog` | 编辑器/IDE/工具配置 |

**规则**：
- 判定核心主题域 → 用对应图标
- 匹配多个 → 用最具体的（Bug 优先于 DB，DB 优先于 通用代码；Postgres / MySQL 类 debug 结论优先用 Bug）
- 都不匹配 → 不传 `--icon`，让 CLI 用类型默认
- **只能用上表里的值**；扩展词表要先改本文件

## 九、CLI 调用模板

**单条 til**：

```bash
cat <<'MD' | ./scripts/trilium.py note til \
    --topic "<Topic>" \
    --title "<title>" \
    --source-session "<sessionId>" \
    --note-date "<YYYY-MM-DD>" \
    --icon "<bx bx-xxx>"
<正文 markdown>
MD
```

**单条 idea / ref**：命令换 `idea` / `ref`；ref 额外加 `--url "<url>"`。

**读现有主题**（每次批量或不确定命名时都跑）：

```bash
./scripts/trilium.py note topics
```

## 十、失败处理

- 每条独立调 CLI，失败继续下一条
- CLI 非零退出 → 先脱敏 stderr，再加入 `failed[]`；无法确认是否脱敏时，用"CLI 调用失败，stderr 已省略以避免泄露敏感信息"代替原文
- 跳过（去重、口径不够）→ 加入 `skipped[]`，写清理由

## 十一、返回格式

严格 JSON：

```json
{
  "notes": [
    {
      "type": "til | idea | ref",
      "title": "...",
      "topic": "...",
      "why": "一句话说明为什么值得记，≤30 字且不重复标题",
      "noteId": "...",
      "url": "http://.../#root/..."
    }
  ],
  "skipped": [
    { "reason": "...", "hint": "..." }
  ],
  "failed": [
    { "attemptedTitle": "...", "error": "..." }
  ],
  "note": "可选：整体说明"
}
```
